# Elixis Agency Website Operations

This repo contains the Elixis Agency public website. The Elixis AI service family includes two primary services, Elixis Collect and Elixis Intake; an optional Elixis Visibility add-on; and higher-touch custom work through Elixis Systems. The AI Audit is the first step used to choose the right workflow. The website is served from `public/` by the same Express app used for the Retell backend, but website files must never contain backend secrets.

## How Deployment Works

- Previous manual deploys can show source `vercel deploy`.
- The current deployment goal is GitHub-based: commit to `main`, let Vercel build from `rizefr/RetellDemo`, then verify `https://elixis.agency`.
- GitHub-connected deploys should show commit and branch information instead of only `vercel deploy`.
- If a GitHub deployment fails with `Could not read package.json` at `/vercel/path0/package.json`, the GitHub repo root is missing the actual project files. Because Root Directory is `./`, `package.json`, `vercel.json`, `public/`, `src/`, and `web/` must be committed at the repository root.
- `Ready` means the deployment built successfully.
- `Error` means the deployment failed.
- `Production` means that deployment is currently assigned to the production domain.
- Short codes such as `HkweHsC9u` are deployment IDs or aliases, not branch names.
- The custom domain `elixis.agency` points to the latest promoted production deployment.

## Root Directory Setting

Vercel Root Directory `./` means Vercel expects the app at the repo root. Because the root is `./`, files like `package.json`, `vercel.json`, `public/`, `src/`, and `web/` should live at the top level of the GitHub repo.

Do not put the whole project inside an extra nested folder unless the Vercel Root Directory setting is changed to match that nested folder.

## Important Website Files

- `public/index.html`: landing page markup and copy.
- `public/demo/index.html`: Elixis Intake demo page. The current demo is pest-control-specific, but the page explains that the intake structure can adapt to other service businesses.
- `public/collections/index.html`: public marketing page for Elixis Collect payment follow-up. Do not confuse this with protected backend `/outbound` tooling.
- `public/ai-seo/index.html`: optional Elixis Visibility / AI search visibility add-on page.
- `public/booking/index.html`: standalone branded Cal.com booking page.
- `public/site-config.js`: public, non-secret website URLs and labels.
- `public/site.js`: mobile menu, section scrolling, Retell modal, config application, reveal animation, and Cal.com embed bootstrapping.
- `web/styles.css`: Tailwind v4 source CSS and custom design system.
- `public/styles.css`: compiled CSS generated by `npm run build:styles`.
- `public/polish.css`: small production-safe override layer for the focused landing-page polish.
- `public/images/`: Elixis logo assets.
- `public/server.ts`: Vercel Express entrypoint. Do not remove unless the backend is intentionally moved elsewhere.

## Current Page Structure

The live website is organized around a general business AI agency offer:

1. `/`: general Elixis Agency homepage. It presents Elixis Collect first, Elixis Intake second, Elixis Visibility as an optional add-on, and Elixis Systems as the custom option.
2. `/collections/`: primary Elixis Collect / accounts-receivable follow-up marketing page.
3. `/demo/`: primary Elixis Intake demo page. The current Retell voice demo uses a pest-control scenario as the example industry.
4. `/ai-seo/`: optional Elixis Visibility and AI-style search audit page.
5. `/booking/`: standalone embedded Cal.com booking page and AI Audit entry point.

Elixis Systems uses the existing `/#custom-workflows` homepage anchor rather than a new route. It is the higher-touch option for a specific repetitive workflow that does not fit Collect, Intake, or Visibility.

AI Audit is not a numbered service. It is the first step used to identify whether Collect, Intake, Visibility, or a custom Elixis System is the best workflow to improve first.

The homepage should not be pest-control-specific. It should speak to local service businesses, B2B companies, appointment-based businesses, and companies with customer calls, unpaid invoices, follow-up gaps, or visibility problems. Pest-control-specific copy belongs on `/demo/` because the current Retell demo agent is trained around that example.

Primary booking links open `https://elixis.agency/booking/`, where the Cal.com calendar is embedded. The footer contains the Elixis Agency contact number and both public contact emails.

To update site copy or section order, edit the matching page under `public/`. To update layout and responsive styling, edit `web/styles.css`, then run `npm run build:styles` to regenerate `public/styles.css`.

## Backend Boundaries

Do not change backend code for ordinary website copy/layout updates.

- Do not modify `/outbound`, outbound APIs, payment-collection backend routes, Stripe/webhook routes, Retell backend logic, Supabase logic, or server-side database code for public marketing-page edits.
- The public payment collection marketing page is `/collections/`.
- If a protected backend/admin outbound route exists at `/outbound`, leave it untouched unless the task explicitly asks for backend work.
- Website files must not contain API keys, Supabase keys, Retell keys, Cal.com keys, or `.env` values.

## How to update the Retell demo iframe/link

Use only a public Retell orb URL in this format:

```text
https://agent.retellai.com/orb/AGENT_ID?token=PUBLIC_WIDGET_TOKEN
```

Do not use a Retell API key, dashboard URL, or backend credential.

The public demo URL has one source of truth: `AI_DEMO_ORB_URL` in `public/site-config.js`. The `/demo/` iframe reads that value through `public/site.js`; the Retell URL is not duplicated in the HTML.

### Manual method

1. Open `public/site-config.js`.
2. Replace only the value of `AI_DEMO_ORB_URL` with the complete public Retell orb URL.
3. Keep `AI_DEMO_PHONE` as `tel:+19842075346` unless the demo phone number also changes.
4. From the GitHub repo root, run the checks:

```bash
npm run build
npm run lint
npm test
node --check public/site.js
```

5. Commit the relevant frontend files to `main`.
6. Push `main`; Vercel deploys automatically from GitHub.
7. Open `https://elixis.agency/demo/`, confirm the new orb loads, and verify the page still scrolls on desktop and mobile.

### Safe utility method

The existing utility validates the public Retell orb format, updates `public/site-config.js`, and refreshes the frontend cache tags:

```bash
npm run set:demo-url -- "https://agent.retellai.com/orb/NEW_AGENT_ID?token=PUBLIC_TOKEN"
```

The utility updates:

- `public/site-config.js`: the single public demo URL value
- `public/index.html`: cache tags for `site-config.js` and `site.js`
- `public/demo/index.html`: cache tags for the standalone demo page

It never reads or changes `.env`.

Confirm the public config contains the intended URL:

```bash
rg -n "AI_DEMO_ORB_URL|agent\\.retellai\\.com/orb/" public/site-config.js public/site.js public/demo/index.html scripts/setDemoUrl.mjs
```

Review the exact files that will be committed:

```bash
git diff -- public/index.html public/demo/index.html public/site.js public/site-config.js scripts/setDemoUrl.mjs
git status --short
```

Commit and push to `main`:

```bash
git add public/index.html public/demo/index.html public/site.js public/site-config.js scripts/setDemoUrl.mjs
git commit -m "Update AI demo orb URL"
git push origin main
```

Wait for Vercel to finish the GitHub deployment, then verify:

- Open `https://elixis.agency/demo/`
- Confirm the address bar remains on `elixis.agency/demo/` while the orb loads
- Confirm `Call Demo Agent` still uses `tel:+19842075346`

Local-only testing override:

```text
http://localhost:3000/?demoOrbUrl=https%3A%2F%2Fagent.retellai.com%2Forb%2FNEW_AGENT_ID%3Ftoken%3DPUBLIC_TOKEN
```

This override only works on localhost-style hosts and only affects the current browser session. It does not persist to production files.

Never paste Retell API keys, Cal.com API keys, Supabase keys, or private white-label tokens into frontend files.

## Booking Links

Current public booking defaults live in `public/site-config.js`:

- `BOOKING_URL`: external fallback booking page.
- `CAL_PUBLIC_BOOKING_URL`: public Cal.com event link used by the inline embed.
- `CAL_EVENT_TYPE_ID`: reference ID for the Cal event type.

The inline calendar is hosted on `public/booking/index.html` and is available at:

```text
https://elixis.agency/booking/
```

Primary `Book an AI Audit`, `Book Demo`, and `Book a short call` links should point to `/booking/`. Keep the external Cal.com URL only as a fallback in case the inline calendar cannot load.

If a new Cal event type needs to be resolved, do that server-side or in Cal.com, then put only the public booking URL in `public/site-config.js`.

## Retell Branding Limitation

The visible Elixis page UI says `Talk to AI`. Text inside the Retell-hosted iframe is controlled by Retell and cannot be safely changed from parent-page CSS or JavaScript because the iframe is cross-origin. Use Retell-supported white-label/widget settings if internal iframe branding needs to change.
